What is Pressroom?

Pressroom is a publishing tool for ideas. You write papers in your private Ideas Workbench GitHub repo using Obsidian (or any markdown editor), then use Pressroom to generate a PDF, review it, and publish a versioned snapshot to your public Pressroom Pubs repo.

Pressroom holds no content itself. Everything lives in your GitHub repos. The app just connects the two repos and handles PDF generation.

Basic workflow
  1. Write your paper in Obsidian
    In your Ideas Workbench repo, create a folder named with your paper's slug (e.g. my-paper-idea). Inside it, create my-paper-idea/publish/my-paper-idea.md. This file holds both the YAML frontmatter (title, gate, author, etc.) and the paper body. Write freely — Pressroom won't touch it until you ask it to.
  2. Generate a starting prompt (optional)
    Go to New paper prompt in the sidebar. Enter the slug and pick a template, then click Generate prompt. Copy the result and paste it into Claude, ChatGPT, or any AI assistant. The AI will produce a structured markdown paper with [PLACEHOLDER: ...] markers for anything incomplete. Drop that output into your publish/{slug}.md file.
  3. Load the paper in Pressroom
    Go to QA & Publish. Type the slug into the Load paper box (exactly as the folder is named) and click Load. Pressroom reads the frontmatter from your workbench repo and populates the form.
  4. Work through the QA checklist
    The checklist is a prompt, not a gate. Tick each item as you review it. You can publish without checking everything — the checklist is there to slow you down and make the review deliberate.
  5. Review and edit metadata
    Check the title, subtitle, gate, version, and author fields. Set the Gate to match how complete the paper is — the version string auto-fills from the gate. Fill in prior art notes and AI disclosure if relevant. Click Save metadata to write changes back to your GitHub repo.
  6. Preview the PDF
    Click Preview PDF. Pressroom runs Pandoc + XeLaTeX on your markdown, generates a PDF, pushes a review copy back to your workbench repo (publish/{slug}.pdf), and shows the PDF inline. This can take 10–30 seconds. If something looks wrong, fix the markdown in Obsidian, sync to GitHub, and preview again.
  7. Publish
    When you're happy with the PDF, click Publish → GitHub. Pressroom:

    1. Creates a versioned snapshot in your workbench repo at {slug}/v{version}/ containing the .md, .pdf, and any artifacts.
    2. Mirrors that snapshot to your Pressroom Pubs repo so it's publicly accessible.

    The snapshot is frozen. Future edits and publishes create new versioned folders — old versions are never modified.
Gates — what they mean

A gate is a label for how complete an idea is. It also determines the version string that gets stamped on the snapshot.


Gate Version What it means
Alpha v0.1-alpha Title and stubs only. The idea is captured but not yet developed.
Exploratory v0.1-exploratory Real content present. Thinking is visible and unfinished. Placeholders expected.
Draft v0.2-draft Structured and substantive. Placeholders are intentional, not forgotten.
Review v0.3-review Complete. No unresolved placeholders. Ready for final QA.
Published v1.0 Packaged and live. Final state.
Tip: You can publish at any gate. Publish early and often — an Alpha snapshot is a valid record that an idea existed at a point in time.
What is a slug?

A slug is the URL-friendly name for your paper. It must be all lowercase, with words separated by hyphens and no spaces or special characters.

Examples: my-great-idea, pressroom-spec, ai-research-methods

The slug is also the folder name in your Ideas Workbench repo. Everything must match — the folder name, the .md filename, and the slug you type into Pressroom.

Folder structure: ideas-workbench / {slug} / publish / {slug}.md
Placeholders

Pressroom uses a specific placeholder format throughout your paper to mark content that still needs to be written:

[PLACEHOLDER: description of exactly what goes here]

The description should be specific enough that you (or an AI) know what to write. Vague placeholders like [PLACEHOLDER: content here] are not useful — be precise about what's needed.

At Review and Published gates, all placeholders should be resolved. At earlier gates they are expected and acceptable.

Common issues
PDF preview fails

Usually a LaTeX error caused by special characters or formatting in your markdown. Check for unescaped %, &, _, or # characters outside of code blocks. LaTeX treats these as control characters. The error message in the app will indicate which line caused the problem.

Paper won't load ("not found")

The slug you typed doesn't match the folder name in your workbench repo exactly. Check capitalisation and hyphens. Also confirm the file exists at {slug}/publish/{slug}.md — both the folder and the file must exist.

Metadata changes aren't saving

Click Save metadata before previewing or publishing. The form fields are not automatically synced — you must explicitly save to write changes back to GitHub.

Publish succeeded but I can't see it in pressroom-pubs

GitHub can take a few seconds to reflect new commits. Refresh your pubs repo page. If it's still missing after 30 seconds, check the action message in Pressroom for any error — the snapshot may have succeeded but the mirror step may have failed. You can re-publish to retry.
